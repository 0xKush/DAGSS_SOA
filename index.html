¿Qué es SOA?

  • Estilo arquitectónico para la construcción de aplicaciones software en base a servicios disponibles.

  • Un servicio representa una actividad de negocio del sistema específico.

Origen

  • Evolución de los modelos de desarrollo con el paso de los años.

  • Años 80 - 90: Modelos orientados a objetos y basados en componentes.

  • Aplicaciones fueron desarrolladas sin consideración de interacción con otros sistemas.

Origen

  • Para poder competir y crecer las organizaciones tenían que realizar inversiones masivas en hardware, software y recursos humanos.

  • Unión de los objetivos de negocio con la infraestructura de TI.

Origen

Necesidad de un sistema nuevo
  • Ofrezca ventaja competitiva.
  • Responda de manera efectiva.
  • Reduzca los costes de TI.
  • Flexibilidad ante cambios.
  • Integración.
  • Interconectar procesos, personas e información.

Origen

Aparecen los modelos orientados a servicios.

Ventajas

  • Controlar un problema de forma general.

  • Facilidad para la colaboración con otros gentes.

  • Facilidad para la integración con otras tecnologías.

Ventajas

  • Aplicaciones flexibles.

  • Aplicaciones reutilizables y adaptables a nuevos entornos.

  • Reducción de costes.

Terminología

Servicio
  • Una función sin estado, auto-contenida, que acepta una llamada y devuelve una respuesta.

  • No depende del estado de otras funciones o procesos.

Terminología

Orquestación
  • Secuenciar los servicios y proveer la lógica adicional para procesar datos.

Terminología

Sin estado
  • No mantiene ni depende de condición pre-existente alguna.

  • En una SOA los servicios no son dependientes de la condición de ningún otro servicio.

  • Reciben en la llamada toda la información que necesitan para dar una respuesta.

Terminología

Proveedor
  • La función que brinda un servicio en respuesta a una llamada o petición desde un consumidor.

Terminología

Consumidor
  • La función que consume el resultado del servicio provisto por un proveedor.

  • Para ofrecer servicios fuera del firewall a nuevos mercados.

  • Para proporcionar análisis en tiempo real de los eventos comerciales.

  • Para acelerar el tiempo de comercialización.

Ciclo de vida

El Ciclo de Vida de SOA consta de 4 actividades principales:
  • Modelado

  • Ensamblado

  • Despliegue

  • Gestión

Ciclo de vida

Modelado
  • Capturar, analizar, simular y optimizar modelos

  • Reducir el riesgo y aumentar la flexibilidad

Ensamblado
  • De componentes nuevos y antiguos

  • Para ejecutar y gestionar procesos de negocio

Ciclo de vida

Despliegue

Gestión
  • Análisis de la información de negocio derivada para tomar acciones coordinadas y “a tiempo”

Diseño

Metodología de modelado y diseño para aplicaciones SOA:
  • Análisis y diseño orientado a servicios.

Diseño

El resultado son sistemas:
  • Con alta reusabilidad

  • Con bajo acoplamiento

  • Poco sensibles a los cambios

Diseño

Permiten modelos de:
  • Consumo de terceros

  • Colaboración

  • Integración de tecnologías

Desarrollo. Capas de software

En SOA se definen las siguientes capas de software:
  • Aplicaciones básicas

  • Capa de exposición de funcionalidades

  • Capa de integración de servicios

  • Capa de composición de procesos

  • Capa de entrega

Desarrollo. Capas de software

Aplicaciones básicas
  • Cualquier arquitectura o tecnología

  • Sistemas geográficamente dispersos

Capa de exposición de funcionalidades
  • Las funcionalidades de la capa aplicativa son expuestas en forma de servicios (servicios web).

Desarrollo. Capas de software

Capa de integración de servicios
  • Facilitan el intercambio de datos entre elementos de la capa aplicativa orientada a procesos empresariales.

Capa de composición de procesos
  • Define el proceso en términos del negocio y sus necesidades, y que varía en función del negocio.

Desarrollo. Capas de software

Capa de entrega
  • Encargada de que los servicios se desplieguen a los usuarios finales.

SOA y los servicios web

Arquitectura orientada a servicios
  • Servicios residentes en Internet o en una intranet

  • Servicios web no son requisito para implementar SOA, aunque sí lo más común.

SOA y los servicios web

Servicios Web
  • Tecnología de llamada a procedimientos remotos (RPC)

Características
  • Uso de las tecnologías y protocolos Web

  • Fuertemente basado en XML

  • Independiente del lenguaje de programación.

SOA y los servicios web

Estándares en servicios web
  • XML
  • HTTP
  • SOAP
  • REST
  • WSDL
  • UDDI

Un sistema SOA no necesariamente utiliza estos estándares para ser "Orientado a Servicios"

4. Cliente del Servicio

public class TestClient {

	public static void main(String[] args) throws MalformedURLException {

		// Servicio Cálculo de Muerte
		URL deathUrl = new URL("http://localhost:8080/getDeath?wsdl");
		// Hacemos referencia al servicio
		QName name = new QName("http://ejemploServicio/",
					"DeathServiceService");

		// Creamos cliente del servicio con acceso a los puertos remotos
		Service deathService = Service.create(deathUrl, name);

		ServiceInterface deathServiceInterface = 
					deathService.getPort(ServiceInterface.class);

		// Llamamos al servicio
		System.out.println(deathServiceInterface.getResponse("Pepito"));
	}
